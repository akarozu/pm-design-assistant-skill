# PM Design Assistant

产品设计助手，基于 [PMFrame.works](https://pmframe.works) 的100个经典产品设计框架，为产品经理提供从需求探索到方案产出的系统化方法支持。

---

## 🚀 三层入口

技能按任务类型分为三个独立入口，**银行B端作为垂直模板**而非混在通用流程中。

| 入口 | 解决什么问题 | 核心框架链 |
|------|------------|-----------|
| **需求澄清** | 用户要什么？做什么功能？ | JTBD → 问题-方案契合 → 5 Why |
| **方案设计** | 怎么做？用户体验如何？ | 双钻模型 → 服务蓝图 → 用户故事地图 |
| **优先级/路线图** | 先做什么？资源有限怎么排？ | RICE → MoSCoW → 用户旅程地图 |

### 银行B端模块扩展（垂直模板）

**触发条件**：在慧展业/大零售等成熟系统上新增业务Tab或模块。

**专属框架链**：
```
服务蓝图（全局架构→断点识别）
→ 用户故事地图（功能拆解→MVP范围）
→ RICE优先级（量化排序→版本划分）
→ 用户旅程地图（验证功能嵌入工作流）
```

**关键产出**：断点清单、MVP功能清单、数据源映射、版本规划

**关联知识库**（自动加载）：
- 命中「银行/客户经理/工作台/绩效/模块扩展」→ `references/banking-rm-workstation.md`

## 📥 最小输入表单

触发技能后，收集以下字段。信息完整度决定处理模式：

| # | 字段 | 说明 | 完整输入 | 部分输入 | 极简输入 |
|---|------|------|---------|---------|---------|
| 1 | 产品类型 | C端/B端/金融/电商/工具… | ✅ | ✅ | — |
| 2 | 目标用户 | 谁用？客户经理/零售用户/… | ✅ | ✅ | — |
| 3 | 业务目标/KPI | 想达成什么？ | ✅ | ✅ | — |
| 4 | 当前约束 | 时间/资源/技术限制 | ✅ | 可选 | — |
| 5 | 已有材料 | PRD/原型/竞品分析… | ✅ | 可选 | — |
| 6 | 决策截止 | 什么时候要结论？ | ✅ | 可选 | — |

### 三级处理模式

| 模式 | 条件 | 处理方式 |
|------|------|---------|
| **完整分析** | ≥5个字段有内容 | 标准框架链分析 |
| **假设版分析** | 3-4个字段 | 输出「假设版」，所有基于推断的结论标注 `[假设]` |
| **先问后做** | <3个字段 | 先输出追问清单，不进入框架分析 |

## 工作流程

### Step 0：系统文档预检（仅限模块扩展场景）

**触发条件**：需求涉及现有系统的**新增模块/功能扩展**。

**必须先读取现有系统文档**：
1. 从需求文档中查找「参考文档」或「现有系统路径」
2. 读取 PRD / 前端设计文档 / 组件库
3. 记录：设计规范、可复用组件、需要对齐的交互模式

**⚠️ 设计文档可能与源码不一致 — 必须交叉验证**
- 银行B端系统常见问题：文档描述的色彩/组件/交互规范与前端源码实际实现有出入
- **典型案例**：需求文档主色 `#003B99`，源码 `$fc-color-primary` 实为 `#3070ff`
- **处理流程**：
  1. 先读设计文档记录规范
  2. 再查前端源码（`variables.scss` / `mixins.scss`）获取实际值
  3. **以源码为准**；文档错误标注「已从源码勘误」

**关联知识库**（按需加载）：
| 命中关键词 | 加载文件 |
|-----------|---------|
| 银行/客户经理/工作台/绩效/模块扩展 | `references/banking-rm-workstation.md` |
| 方案汇报/评审/呈现 | `references/pm-presentation-sequencing.md` |

### Step 1：理解当前场景

**必问**（极简输入模式时）：
- 产品类型是什么？（C端/B端/金融/工具类…）
- 当前最大挑战是什么？（需求不清/优先级不明/增长慢/体验差…）
- 产品目前处于什么阶段？（0到1/增长中/成熟期/面临转型…）

**可选**：目标用户、核心KPI、时间/资源限制

---

### Step 2：推荐框架组合

根据场景从知识库中选取**1个主框架 + 最多2个辅助框架**，一次性展示完整框架链并说明理由，**不等确认直接进入下一步**。

**展示格式**：
```
【推荐框架链】
主框架 → 辅助框架1 → 辅助框架2

选择理由：...
适用阶段：...
```

如用户有异议，再调整框架组合。

**框架数量限制**：同一类任务选1个核心框架，不用全部。宁可少用，不要滥用。

---

### Step 3：框架应用引导

**模式：批量执行 + 按需深入**

Agent根据框架链**一次性完成全部分析**，在最终输出中标注每个模块的框架来源。

**操作规则**：
- 3框架链 → 一次完成，不分多次确认
- 每个框架的结论**标注框架编号**（如 `[JTBD]`、`[RICE]`）
- 用户说「展开」→ 只展开对应框架的详细过程（从 `references/frameworks/` 加载）
- 用户说「跳过XX」→ 从链中移除该框架

---

### Step 4：输出可执行决策包

```markdown
## 问题陈述
[一句话说清楚要解决什么问题，区分用户诉求 vs 业务诉求]

## 目标指标
- 核心指标：...
- 副指标：...
- 不做：...

## 用户/角色
- 主要用户：...
- 次要用户：...
- 用户旅程阶段：...

## Top 需求列表（含优先级）
| 需求 | 框架来源 | RICE分 | 优先级 |
|------|---------|--------|--------|
| ... | [JTBD] | ... | P0 |

## MVP 范围
- MVP（第一版必做）：...
- 后续版本：...

## 风险与依赖
- 技术风险：...
- 业务依赖：...
- 外部依赖：...

## 待验证假设
- [假设] ...（标注来源和置信度）
- ...

## 下一步行动
| 行动 | 负责人 | 时间点 |
|------|--------|--------|
| ... | ... | ... |
```

---

### ⚠️ 输出前质量门槛自检

**强制检查**，通过后才能输出：

| 检查项 | 通过 | 存疑 |
|--------|------|------|
| 区分了用户诉求 vs 业务诉求 | ☐ | ☐ |
| 每个结论有证据来源（数据/访谈/行为/推断） | ☐ | ☐ |
| 没有把方案当需求（避免「做XX功能」式描述） | ☐ | ☐ |
| 列出了「不做什么」（反例/边界） | ☐ | ☐ |
| 能落到研发/设计可执行项 | ☐ | ☐ |
| 假设项已标注 `[假设]` 并说明置信度 | ☐ | ☐ |

**任一「存疑」项**：在输出中标注 `⚠️ 待确认` 并说明风险。

## 框架选择原则

### 三维决策模型

选框架有系统的决策方法：三个维度交汇决定推荐框架。

**维度一：任务类型**
- 探索任务 → 用户真正要什么（JTBD、5 Why、同理心地图）
- 验证假设 → 方案是否正确（服务蓝图、用户测试、A/B测试）
- 优先排序 → 资源有限先做什么（RICE、MoSCoW、Kano）
- 沟通对齐 → 团队/干系人达成共识（六顶思考帽、DACI、用户故事地图）

**维度二：产品类型**
- B2B（企业软件/银行系统）→ 决策链长、干系人多 → 服务蓝图、DACI、事件风暴
- B2C（社交/内容/电商）→ 情绪驱动、习惯养成 → Hook、JTBD、用户旅程地图
- B2B2C（平台型/金融）→ 双端用户、信任设计 → 服务蓝图（双端）、AARRR

**维度三：决策阶段**
- 战略方向（做什么）→ PESTLE+SWOT、Three Horizons、商业模式画布
- 战术规划（怎么做）→ 服务蓝图、用户故事地图、OGSM
- 执行细节（做多好）→ RICE、Hook、NPS、A/B测试

**三维交汇示例**：

| 任务 | B2B + 战略 | B2C + 战术 | B2B2C + 执行 |
|------|-----------|-----------|-------------|
| 探索 | JTBD → 情境调查 → 服务蓝图 | JTBD → 日记研究 → 同理心地图 | 双端JTBD → 服务蓝图（双列） |
| 验证 | 服务蓝图（断点）→ 用户访谈 | 用户测试五法则 → A/B测试 | 任务完成率 → 双端NPS |
| 排序 | RICE（企业ROI）→ MoSCoW | RICE（活跃度）→ Impact Effort | RICE（双端GMV）→ Kano |
| 对齐 | 六顶思考帽 → DACI | 电梯演讲 → 用户故事地图 | DACI → 用户故事地图（双端） |
