# PM Design Assistant

产品设计助手，基于 [PMFrame.works](https://pmframe.works) 的100个经典产品设计框架，为产品经理提供从需求探索到方案产出的系统化方法支持。

## 核心能力

1. **场景诊断** — 识别当前产品阶段和核心挑战
2. **框架推荐** — 根据场景推荐最适合的框架组合
3. **流程引导** — 引导完成框架的实际应用
4. **方案产出** — 输出结构化的产品方案

## When to Use

- 用户说「分析需求」「做什么功能好」「需求不清晰」「不知道先做哪个」
- 用户说「功能优先级」「怎么排序」「RICE怎么打」
- 用户说「增长慢」「AARRR」「留存低」
- 用户说「从0到1」「PMF」「MVP怎么做」
- 用户说「用户真正要什么」「JTBD」「用户访谈」
- 用户说「竞品分析」「SWOT」「蓝海策略」
- 用户说「框架是什么」「PMF怎么判断」
- **银行B端模块扩展**：在慧展业/大零售等成熟系统上新增业务Tab或模块

**不要用于**：原型设计（用 prototype-design）、文档版本管理（用 requirement-doc-manager）、技术架构设计

## 工作流程

### Step 0（重要）: 检查现有系统文档

**触发条件**：当需求涉及对现有系统的**新增模块/功能扩展**时。

**必须先读取现有系统文档，再进行任何分析**：
- 产品需求文档（PRD）
- 现有前端/后端设计文档
- 现有组件库/设计规范

**操作步骤**：
1. 从需求文档中查找"参考文档"或"现有系统路径"
2. 读取现有系统的设计文档
3. 记录：设计规范、可复用组件、需要对齐的交互模式
4. 再进入 Step 1 分析

**⚠️ 设计文档可能与源码不一致 — 必须交叉验证**
- 银行B端系统常见问题：设计文档（/MD文档）描述的色彩/组件/交互规范，与前端源码（`src/styles/variables.scss`、`src/components/`）实际实现有出入
- **典型案例**：需求文档记录主色 `#003B99`，源码 `$fc-color-primary` 实为 `#3070ff`；间距单位文档写 `px`，源码用 `rpx`
- **处理流程**：
  1. 先读设计文档（PRD/前端文档）记录规范
  2. 再查前端源码（`src/styles/variables.scss` / `src/styles/mixins.scss`）获取实际值
  3. 以源码为准；文档错误在需求文档中标注「已从源码勘误」
- **源码关键文件**：样式变量（`variables.scss`）、页面路由（`pages.json`）、组件源码（`components/`）
- 否则风险：分析结果与现有系统实际规范冲突 → 强制返工

**否则风险**：分析结果与现有系统不一致，强制返工。

> **反面案例**：分析"零售"模块时未读"慧展业"工作台文档 → 提出的优化建议与现有设计系统冲突 → 必须重新分析。

### Step 1：理解当前场景

**必问**：
- 产品类型是什么？（C端/B端/电商/金融/工具类…）
- 当前最大挑战是什么？（需求不清/优先级不明/增长慢/体验差…）
- 产品目前处于什么阶段？（0到1/增长中/成熟期/面临转型…）

**可选**：
- 目标用户是谁
- 核心KPI是什么
- 有没有时间/资源限制

### Step 2：推荐框架组合

根据场景从知识库中选取2-4个最相关的框架，一次性展示完整框架链并说明理由，**不等确认直接进入下一步**。

**展示格式**：
```
【推荐框架链】
框架1 → 框架2 → 框架3

选择理由：...
```

如用户有异议，再调整框架组合。

### Step 3：框架应用引导

**模式：批量执行 + 按需深入**

不再逐个框架暂停确认。Agent根据框架链**一次性完成全部分析**，在最终输出中标注每个模块的框架来源。

用户可随时说「展开第2步」「只看RICE」「深入讲解」，Agent立即补充细节。

**操作规则**：
- 4框架链 → 一次完成，不分4次确认
- 每个框架的结论**标注框架编号**（如 `[JTBD]`、`[RICE]`）
- 用户说「展开」→ 只展开对应框架的详细过程
- 用户说「跳过XX」→ 从链中移除该框架

### Step 4：方案整合输出

```markdown
## 需求洞察
- 用户画像：...
- 核心痛点：...

## 问题定义
- POV陈述：...
- 优先级最高的问题：...

## 解决方案
- Top 3 方案选项：...
- 推荐方案：...

## 验证计划
- 如何验证这个方案：...
- 成功指标：...

## 落地计划
- 下一步行动：...
- 时间线：...
```

## 框架选择原则

### 核心原则：三维决策模型

选框架不是凭感觉，而是有系统的决策方法。三个维度交汇决定推荐框架：

**维度一：任务类型**
- 探索任务 → 用户真正要什么（JTBD、5 Why、同理心地图）
- 验证假设 → 方案是否正确（服务蓝图、用户测试、A/B测试）
- 优先排序 → 资源有限先做什么（RICE、MoSCoW、Kano）
- 沟通对齐 → 团队/干系人达成共识（六顶思考帽、DACI、用户故事地图）

**维度二：产品类型**
- B2B（企业软件/银行系统）→ 决策链长、干系人多 → 服务蓝图、DACI、事件风暴
- B2C（社交/内容/电商）→ 情绪驱动、习惯养成 → Hook、JTBD、用户旅程地图
- B2B2C（平台型/金融）→ 双端用户、信任设计 → 服务蓝图（双端）、AARRR

**维度三：决策阶段**
- 战略方向（做什么）→ PESTLE+SWOT、Three Horizons、商业模式画布
- 战术规划（怎么做）→ 服务蓝图、用户故事地图、OGSM
- 执行细节（做多好）→ RICE、Hook、NPS、A/B测试

**三维交汇示例**

| 任务 | B2B + 战略 | B2C + 战术 | B2B2C + 执行 |
|------|-----------|-----------|-------------|
| 探索 | JTBD → 情境调查 → 服务蓝图 | JTBD → 日记研究 → 同理心地图 | 双端JTBD → 服务蓝图（双列） |
| 验证 | 服务蓝图（断点）→ 用户访谈 | 用户测试五法则 → A/B测试 | 任务完成率 → 双端NPS |
| 排序 | RICE（企业ROI）→ MoSCoW | RICE（活跃度）→ Impact Effort | RICE（双端GMV）→ Kano |
| 对齐 | 六顶思考帽 → DACI | 电梯演讲 → 用户故事地图 | DACI → 用户故事地图（双端） |

### 宁可少用，不要滥用
- 同一类问题选1个核心框架，不用全部
- 优先选用户已理解的框架，降低认知成本

### 按顺序推进
- 先用户洞察，再问题定义，然后创意，最后验证
- 跳跃顺序会导致返工

### 数据+直觉并重
- 定量框架（A/B测试、RICE）和定性框架（访谈、头脑风暴）组合使用
- 不要纯靠数据，也不要纯靠直觉

## 精选框架详解

### #44 RICE优先级模型

**公式**：R × I × C ÷ E = RICE Score

| 因子 | 定义 | 评分标准 |
|------|------|---------|
| Reach（触达） | 每周期受影响用户数 | 绝对数量 |
| Impact（影响） | 对核心指标的影响 | 0.25最小 / 0.5低 / 1中 / 2高 / 3极高 |
| Confidence（信心） | 估算的可靠程度 | 100%数据 / 80%推断 / 50%猜测 |
| Effort（工作量） | 所需人月数 | 设计+工程+测试加总 |

**使用时机**：
- 季度规划时的功能优先级排序
- 多产品线的资源分配决策
- 团队对优先级有强烈分歧时

**五条原则**：
1. 先对齐Reach的定义单位（每周/月/季度）
2. Impact分数要和具体指标挂钩
3. Confidence低于50%时先做小实验
4. Effort用相对单位（人月）不用绝对天数
5. RICE分数是对话起点不是终点

### #57 AARRR海盗指标

**五层漏斗**：

| 阶段 | 核心问题 | 关键指标 |
|------|---------|---------|
| A - Acquisition | 用户从哪些渠道发现你？ | CPA、渠道转化率 |
| A - Activation | 用户第一次体验好吗？ | 核心动作完成率 |
| R - Retention | 用户会回来吗？ | D1/D7/D30留存率 |
| R - Referral | 用户会介绍给别人吗？ | NPS、K-factor |
| R - Revenue | 用户愿意付费了吗？ | ARPU、LTV |

**Airbnb案例启示**：
- 2009年问题不在获客，在激活
- 仅30%注册房东实际发布房源
- 解决方案：雇摄影师免费拍高质量照片
- 结果：NYC预订量提升2倍以上

**五条原则**：
1. 从留存开始优化，而不是获客
2. 每阶段只设一个最关键指标
3. 先找到"啊哈时刻"再优化激活
4. 不要同时解决所有漏斗问题
5. Referral是产品体验的终极测验

### #59 Hook上钩模型

**核心四步循环**：触发→行动→可变奖励→投入

| 步骤 | 关键洞察 |
|------|---------|
| Trigger（触发） | 外部触发→内部触发（无聊/焦虑/孤独）是终极目标 |
| Action（行动） | B=M×A×T；动作越简单，完成率越高 |
| Variable Reward（可变奖励） | 不可预测性创造渴望；社会/猎物/自我三类奖励 |
| Investment（投入） | 用户存入"价值"（数据/内容/关系）→加载下次触发弹药 |

**何时用**：高频使用产品（社交/游戏/内容平台）。低频产品（报税软件）强行套用适得其反。

**真实案例**：Instagram 2010-2018，日均使用53分钟——完全由习惯循环驱动而非广告预算。新用户第一张照片发出后获得点赞通知（投入→触发），数周后内部触发形成。

**责任提醒**：Nir Eyal 2019年《Indistractable》反思Hook伦理——习惯设计是工具，不等于有义务不顾后果使用。

**D7留存**是习惯形成的关键指标，而非D1。**不使用时的不适感**是最强信号。

### #17 双钻模型

**四阶段流程**（英国设计委员会2005年提出）：

```
钻石一：问题空间          钻石二：方案空间
Discover(发散) → Define(收敛) → Develop(发散) → Deliver(收敛)
   广泛探索用户研究          针对已定义问题发散创意
   用户访谈/影子观察         快速原型/用户测试
   收敛：HMW问题句          收敛：最优方案上线
```

**最关键洞察**：跳过第一颗钻石（直接跳到解决方案）是最高频错误。NHS急诊室改造案例——原始问题「如何缩短等待时间」，经Discover阶段发现真问题是「患者不知道自己在等什么」，最终低成本看板+护士主动沟通脚本让满意度提升28%，而等待时间几乎没变。

**产出区分**：
- Define阶段输出**HMW问题句**（不是需求文档）
- Develop阶段要求**至少20个方案**才开始筛选

### #60 PMF框架

**三种测量方法**：

**方法1·Sean Ellis 40%测试**（最常用）：
> "如果你无法再使用这个产品，你会有什么感受？"
> - 非常失望 ≥40% → PMF达成
> - 非常失望 30-40% → 接近，需调优
> - 非常失望 <30% → Pivot信号

**方法2·留存曲线对比**：
