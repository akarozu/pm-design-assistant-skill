# #04 JTBD（用户想要完成的任务）

## 核心概念

用户不是买产品，而是"雇佣"产品来完成生活中的一个**任务**。

> "我买钻头不是要买钻头，而是要买墙上那个洞。"

## JTBD陈述格式

```
当 [情境]，我想要 [动机]，以便达成 [目标]。
```

## 使用时机

- 不知道做什么功能（需求不清）
- 探索用户真正要什么
- 从功能导向转向用户导向

## 与功能需求的区别

| 功能需求（避免） | JTBD需求（推荐） |
|----------------|-----------------|
| 添加"收藏"按钮 | 在离开页面后能轻松找到之前浏览的内容 |
| 增加"消息提醒" | 不错过重要的合作机会通知 |
| 开发"数据分析"模块 | 知道我的客户最近在关心什么 |

## 案例

**功能视角**：给银行客户经理做一个"客户详情"页面，展示基本信息、持有产品、联系记录。

**JTBD视角**：
- "当我想给客户推荐产品时，我希望快速知道这个客户最关注什么（收益/安全/流动性），以便给出最适合的建议。"
- "当客户表现出流失迹象时，我希望提前知道，以便主动联系挽留。"

## ⚠️ 不适用条件（B端/监管场景必须先读）

| 场景 | 风险 | 替代框架 |
|------|------|---------|
| **B端审批流** | 审批流的任务导向是"完成审批流程"而非"完成任务"，JTBD的"雇佣"逻辑不适用 | Kano模型（必备型/期望型） |
| **强制使用场景** | 用户不是"雇佣"产品而是"被迫使用"，行为动机分析框架失效 | 服务蓝图（流程视角） |
| **合规强制场景** | 监管要求的功能不存在"用户想要"的空间 | 直接用功能清单+RICE |
| **探索阶段用户说不清楚任务** | 用户无法清晰表达（可用用户访谈先行） | 用户访谈五问法 |

> **B端审批流判断标准**：如果用户使用产品的目的是「完成上级分配的审批任务」，且使用行为由组织流程驱动而非个人需求驱动，则判定为B端审批流，改用Kano模型。
