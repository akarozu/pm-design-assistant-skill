# #04 JTBD（用户想要完成的任务）

## 核心概念

用户不是买产品，而是"雇佣"产品来完成生活中的一个**任务**。

> "我买钻头不是要买钻头，而是要买墙上那个洞。"

## JTBD陈述格式

```
当 [情境]，我想要 [动机]，以便达成 [目标]。
```

## 使用时机

- 不知道做什么功能（需求不清）
- 探索用户真正要什么
- 从功能导向转向用户导向

## 与功能需求的区别

| 功能需求（避免） | JTBD需求（推荐） |
|----------------|-----------------|
| 添加"收藏"按钮 | 在离开页面后能轻松找到之前浏览的内容 |
| 增加"消息提醒" | 不错过重要的合作机会通知 |
| 开发"数据分析"模块 | 知道我的客户最近在关心什么 |

## 案例

**功能视角**：给银行客户经理做一个"客户详情"页面，展示基本信息、持有产品、联系记录。

**JTBD视角**：
- "当我想给客户推荐产品时，我希望快速知道这个客户最关注什么（收益/安全/流动性），以便给出最适合的建议。"
- "当客户表现出流失迹象时，我希望提前知道，以便主动联系挽留。"

## 不适用条件

- **B端审批流**：任务导向是"完成审批"而非"完成任务"，Kano更合适
- 高度监管强制场景（如合规报表）
- 探索阶段用户说不清楚自己的任务（先用用户访谈）
