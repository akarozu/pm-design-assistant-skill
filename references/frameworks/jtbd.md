# #04 JTBD（用户想要完成的任务）

## 核心概念

用户不是买产品，而是"雇佣"产品来完成生活中的一个**任务**。

> "我买钻头不是要买钻头，而是要买墙上那个洞。"

## JTBD陈述格式

```
当 [情境]，我想要 [动机]，以便达成 [目标]。
```

## 使用时机

- 不知道做什么功能（需求不清）
- 探索用户真正要什么
- 从功能导向转向用户导向

## 与功能需求的区别

| 功能需求（避免） | JTBD需求（推荐） |
|----------------|-----------------|
| 添加"收藏"按钮 | 在离开页面后能轻松找到之前浏览的内容 |
| 增加"消息提醒" | 不错过重要的合作机会通知 |
| 开发"数据分析"模块 | 知道我的客户最近在关心什么 |

## 案例

**功能视角**：给银行客户经理做一个"客户详情"页面，展示基本信息、持有产品、联系记录。

**JTBD视角**：
- "当我想给客户推荐产品时，我希望快速知道这个客户最关注什么（收益/安全/流动性），以便给出最适合的建议。"
- "当客户表现出流失迹象时，我希望提前知道，以便主动联系挽留。"

## ⚠️ 适用性边界（B端/监管场景必须先读）

### ✅ 适用子场景

JTBD 适用于 B 端场景中**用户有自主选择空间**的情况：

- **识别角色任务目标**：用户为完成自己的业务目标而"雇佣"工具（如客户经理为达成业绩目标而选择使用哪些功能）
- **跨角色协作痛点**：识别审批前后（如贷前调查与贷后管理）的衔接痛点，用户目标是"完成业务"而非"完成审批节点"
- **用户想达成的业务结果**：用户对结果负责，有动机优化完成方式（如客户经理想"知道客户最近关心什么"以便推荐产品）

### ❌ 不适用子场景

- **强制合规动作**：监管规定必须执行的操作，用户无自主选择（如监管要求的合规审核流程）
- **纯审批节点流转**：用户目的是"完成上级分配的审批任务"，行为由组织流程驱动而非个人需求驱动
- **监管要求功能**：监管明确要求的功能，用户动机来源于合规而非业务价值
- **用户无自主选择空间**：系统强制要求用户执行，无替代方案或优化动机

### 🔄 条件化判断标准

**JTBD 适用 B 端审批流的前提**：用户是「完成任务」而非「完成审批」。

判断方法：
- 用户是否对业务结果负责？→ 是 → JTBD 可能适用
- 用户的核心动机是「完成组织分配的流程节点」还是「达成自己的业务目标」？→ 「业务目标」 → JTBD 可能适用
- 用户是否有优化完成方式的自主空间？→ 有 → JTBD 可能适用

### 替代框架

| 场景 | 替代框架 |
|------|---------|
| 审批流（用户完成审批任务） | Kano模型（必备型/期望型） |
| 强制使用/合规场景 | 服务蓝图（流程视角）、功能清单+RICE |
| 跨角色协作需要决策者明确 | DACI决策框架 |
| 角色目标不清晰需探索 | 用户访谈五问法 |
