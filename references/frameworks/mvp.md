# #32 精益MVP

## 核心概念

MVP = Minimum Viable Product（最小可行产品）

不是"做最少的功能"，而是**用最小的成本验证最大的风险假设**。

## 三步循环

```
构建（Build）→ 度量（Measure）→ 学习（Learn）
     ↑___________________________________↓
```

## 使用时机

- 从0到1（0到1PMF验证）
- 不确定某个功能是否有人要
- 资源有限需要快速验证

## Buffer案例

- 创始人上线只有两个按钮的落地页
- 第一周120人注册——需求验证成功才构建产品
- 关键洞察：如果Canvas有格子填不上，产品就不该构建

## MVP不是二流产品

MVP的目的是验证假设，不是交付一个将就的产品。

**常见误解**：
- ❌ MVP = 丑陋的半成品
- ✅ MVP = 用最小成本回答关键商业假设

## 关键问题：在做MVP之前

1. **你要验证的核心假设是什么？**
2. **最小的实验是什么？**（不是最小产品，是最小实验）
3. **如何度量成功？**

## 不适用条件

- **高监管不可快速迭代场景**：金融、医疗产品不能快速迭代
- **B端销售周期长**：销售需要完整功能演示
- **已过PMF阶段**：需要的是增长不是验证
